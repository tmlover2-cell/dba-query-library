# SQL 쿼리 라이브러리 MVP PRD

## 핵심 정보

**목적**: DBA가 자주 사용하는 SQL 쿼리를 Notion DB에 저장하고, 웹 인터페이스에서 빠르게 검색/복사할 수 있는 내부 도구
**사용자**: SQL 쿼리를 반복적으로 작성하고 관리하는 DBA (1인 사용자, 인증 불필요)

> **⚠️ 기술 제약**: Notion Rich Text property는 최대 **2,000자** 제한이 있습니다. 2,000자를 초과하는 SQL 쿼리는 저장 불가하므로, Notion에 데이터 입력 시 유의하세요.

## 사용자 여정

1. 쿼리 라이브러리 홈 (메인 페이지)
   아래 액션 중 선택

   [키워드 입력] → URL searchParams 업데이트 → 서버에서 필터링된 목록 반환 (debounce 300ms 적용)
   [DBMS 선택] → URL searchParams 업데이트 → 해당 DBMS 쿼리만 필터링
   [태그 클릭] → URL searchParams 업데이트 → 해당 태그 쿼리만 필터링
   아래

2. 쿼리 카드 목록 (동일 페이지 내 렌더링)
   쿼리 카드 클릭

3. 쿼리 상세 모달 또는 상세 페이지
   복사 버튼 클릭

4. 클립보드 복사 완료 → 토스트 알림 표시 → 계속 검색

## 데이터 모델

### SqlQuery (Notion DB 매핑 모델)

| 필드        | 설명                    | 타입/관계                                                  | Notion Property 타입 |
| ----------- | ----------------------- | ---------------------------------------------------------- | -------------------- |
| id          | Notion 페이지 고유 ID   | string (UUID)                                              | - (자동 생성)        |
| title       | 쿼리명                  | string                                                     | **Title**            |
| sql         | SQL 코드 (최대 2,000자) | string                                                     | **Rich Text**        |
| dbms        | DBMS 종류               | `"MySQL" \| "PostgreSQL" \| "Oracle" \| "MSSQL" \| string` | **Select**           |
| tags        | 멀티셀렉트 태그 목록    | string[]                                                   | **Multi-select**     |
| description | 쿼리 설명               | string                                                     | **Rich Text**        |

> **⚠️ 중요**: SQL 코드는 Notion의 Code block(page body)이 아닌 **Rich Text property**로 저장합니다.
> Code block 방식은 쿼리마다 추가 API 호출이 필요해 Rate Limit(초당 3회)에 충돌합니다.
> Rich Text property 방식은 목록 조회 1회 API 호출로 모든 데이터를 가져올 수 있습니다.
>
> Notion API 응답을 위 구조로 정규화하는 어댑터 함수 필요

## URL 구조

| URL 패턴                                  | 설명                                |
| ----------------------------------------- | ----------------------------------- |
| `/`                                       | 메인 페이지 (쿼리 목록)             |
| `/?q={키워드}`                            | 키워드 검색 결과                    |
| `/?dbms={DBMS명}`                         | DBMS 필터 적용                      |
| `/?tag={태그명}`                          | 태그 필터 적용                      |
| `/?q={키워드}&dbms={DBMS명}&tag={태그명}` | 복합 필터 (중복 적용 가능)          |
| `/queries/[id]`                           | 쿼리 상세 페이지 (Notion 페이지 ID) |

| 대상           | 전략                                | 설명                                                     |
| -------------- | ----------------------------------- | -------------------------------------------------------- |
| 쿼리 목록 조회 | `unstable_cache` (revalidate: 60초) | Notion API Rate Limit(초당 3회) 대응, 60초 간격으로 갱신 |
| 쿼리 상세 조회 | `unstable_cache` (revalidate: 60초) | 동일                                                     |
| 태그 목록      | 쿼리 목록과 동일 캐시 활용          | Notion DB schema Multi-select 옵션에서 추출              |

> Next.js 15는 fetch 기본 캐시가 `no-store`로 변경됨. 반드시 `unstable_cache` 또는 `revalidate` 옵션을 명시해야 합니다.

## 기술 스택

### 프론트엔드 프레임워크

- **Next.js 15.5.3** (App Router + Turbopack) - React 풀스택 프레임워크
- **TypeScript 5** - 타입 안전성 보장
- **React 19.1.0** - UI 라이브러리

### 스타일링 & UI

- **TailwindCSS v4** (설정 파일 없는 새로운 CSS 엔진) - 유틸리티 CSS 프레임워크
- **shadcn/ui** (new-york style) - 고품질 React 컴포넌트 라이브러리
- **Lucide React** - 아이콘 라이브러리 (복사 아이콘 등)

### 코드 하이라이팅

- **Shiki** - SQL 구문 하이라이팅 (Server Component에서 직접 사용 가능, 번들 크기 최소, react-syntax-highlighter 대비 권장)

### 검색 & 필터링

- **URL searchParams** (Native) - 검색/필터 상태를 URL에 저장, App Router 서버 컴포넌트 연동
- **debounce** (직접 구현 또는 `use-debounce` 라이브러리) - 키워드 입력 시 300ms 지연 후 URL 업데이트

> React Hook Form + Zod는 단순 검색 폼에 불필요하며 서버 컴포넌트 구조와 충돌하므로 제외

### 백엔드 & 데이터 소스

- **Notion API** (`@notionhq/client`) - CMS 백엔드, 쿼리 데이터 저장소
- Next.js Server Components / Route Handlers - API 키 보호 및 서버 사이드 패칭

### 배포 & 호스팅

- **Vercel** - Next.js 15 최적화 배포 플랫폼

### 패키지 관리

- **npm** - 의존성 관리
